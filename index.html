The Origin

My curiosity started around 1989. I grew up pulling things apart: computers, networks, and anything that had signals, moving parts, or hidden complexity. I'm happiest when I can trace a problem to its root cause — down to packets, logs, timing, and constraints.

Over time that curiosity turned into a career: enterprise infrastructure, point-of-sale ecosystems, and consulting work where “it has to work everywhere, every day” is the baseline — not the goal.

Principle: If I can't explain it simply, I don't understand it well enough.

The Work

I build and harden systems for organizations that need real-world reliability — not demo-ware. That usually means secure configuration, practical monitoring, migration planning, and automation that keeps things boring (in the best way).

  • Security hardening for servers, networks, and applications — with clear documentation.
  • Infrastructure engineering: Linux, web hosting, DNS, identity, and “it all connects” troubleshooting.
  • Automation that removes repeated manual work (and makes the remaining work auditable).
  • Compliance support (e.g., NIST-style thinking): controls, evidence, and pragmatic implementation.

I also like writing internal runbooks, because the only thing worse than a fragile system is a fragile system nobody can operate.

The Lab

Side projects are where I go deeper: networking, security research, performance tuning, and space systems concepts. I'm not chasing novelty — I'm chasing understanding.

  • LATTICE (NASA L'SPACE NPWEE): Led a proposal effort exploring LunaNet-aligned lunar communications concepts and systems thinking under real program constraints.
  • PRM / “Mnemosync” concept: A self-hosted relationship manager focused on privacy, long-term recall, and data ownership.
  • CTF / security research tooling: Python-first scripts that interrogate APIs, validate assumptions, and reduce noisy workflows to repeatable steps.
  • Minimal web stack: Static-first sites, performance budgets, and “no runtime dependencies” where possible.

If you're into space networking, DTN, or resilient architectures, we'll probably get along.

Bias: Own your data. Keep the stack small. Make failure modes obvious.

The Values

I'm opinionated about a few things — mostly because experience forces it.

  • Simplicity: Less surface area, fewer surprises.
  • Observability: If you can't measure it, you can't operate it.
  • Repeatability: If it isn't reproducible, it isn't solved.
  • Integrity: Clear tradeoffs, no hand-waving.

The Now

Right now I'm building a minimalist personal site, continuing security work, and refining a long-term personal toolkit: better automation, better documentation, and better systems thinking.

I'm also constantly learning — from low-level networking to space systems constraints — because the fun part is connecting the pieces.
